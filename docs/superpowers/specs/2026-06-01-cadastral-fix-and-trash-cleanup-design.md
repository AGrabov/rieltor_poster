## Завдання 1 — Починка пошуку кадастрового номера

### Проблема

`kadastr.live` **остаточно мертвий** — домен більше не резолвиться через DNS
(`getaddrinfo failed`, не просто connection refused). Він є основним джерелом
(кроки 1–3 у `lookup_cadastral_number`), тому кожен об'єкт витрачає 3 завідомо
провальні запити (~6 с) перед тим, як дійти до fallback. Fallback
`kadastrova-karta.com` теж нестабільний: у тестах 2026-06-01 відповідав то
ReadTimeout (>10 с), то `500`.

### Рішення

Нова ланцюжок джерел (від найкращого до fallback) у
[cadastral_lookup.py](../../../crm_data_parser/cadastral_lookup.py):

1. **Крок 0 (без змін)** — regex `_CADNUM_IN_TEXT_RE` по опису/нотатках.
   Кадастр часто вже присутній у тексті.
2. **zem.center — NEW, основне джерело.** Чистий JSON API.
3. **kadastrova-karta.com** — наявний HTML-скрейпер, понижений до fallback.
4. **kadastr.live — видаляється повністю.**

### zem.center API (розвідано 2026-06-01)

```
GET https://api.zem.center/api/search?q=<місто вулиця будинок>&size=20
Accept: application/json

→ 200 {"items": [
    {"cadnum": "8000000000:75:214:0010",
     "address": "м.Київ, Голосіївський р-н, вулиця Львівська, 19",
     "category": "...", "ownership": "...", "area": 0.0992, "area_unit": "га"},
    ...
  ]}
```

Перевірено: запит `q="Київ Львівська 19"` повернув 3 кандидати (19-а, 19, 19-і),
серед них точний `19` → `8000000000:75:214:0010`. Формат `cadnum` збігається з
наявним `_CADNUM_RE`. Це джерело даних — те саме офіційне (`e.land.gov.ua`),
але через зручний JSON-проксі.

### Зміни в коді (`crm_data_parser/cadastral_lookup.py`)

- **Новий хелпер матчингу будинку** `_pick_by_house(candidates, house)`, де
  `candidates` — список `(cadnum, address_text)`. Логіка вибору:
  1. Точний збіг: номер будинку в кінці адреси, не продовжений літерою/дефісом
     (`19`, а не `19-а`). Пріоритет.
  2. Якщо точного немає — варіант із суфіксом (`19-а`).
  3. Якщо нічого не зматчилось — перший валідний `cadnum`.
  Переписує наявну ad-hoc логіку з `_best_cadnum` і
  `_search_kadastrova_karta` у спільний хелпер, який використовують обидва
  джерела.
- **Нова `_search_zem_center(query, house) -> str | None`:** GET JSON,
  зібрати `(cadnum, address)` з валідним `cadnum`, передати в `_pick_by_house`.
  Timeout `(5, 12)`, один retry, помилки → `logger.debug`, повертає `None`.
- **Переписати `lookup_cadastral_number`:** новий порядок кроків:
  1. zem.center: `місто + вулиця + будинок`
  2. zem.center: `місто + вулиця`
  3. kadastrova-karta: `місто + вулиця + будинок`
  4. kadastrova-karta: `місто + вулиця`
- **Видалити:** `_search_raw`, `_SEARCH_URL`, `_HEADERS`, `_best_cadnum`
  (kadastr.live). `_CADNUM_RE`, `_CADNUM_IN_TEXT_RE`, `_strip_street_prefix`,
  `_STREET_PREFIX_RE` зберігаються.
- `_search_kadastrova_karta` зберігається, але переходить на спільний
  `_pick_by_house` замість власної регексп-логіки.

Публічні сигнатури `lookup_cadastral_number`, `enrich_offer_data_with_cadastral`,
`fill_missing_cadastral_numbers` **не змінюються** — інтеграція з `main.py`,
`html_parser.py`, дашбордом лишається тією самою.

### Тести

`tests/test_cadastral_lookup.py` (новий або доповнення):

- `_pick_by_house`: точний збіг пріоритетніший за суфікс; fallback на перший
  валідний; порожній house → перший валідний.
- `_search_zem_center`: на зафіксованому JSON-фікстурі (без мережі, через
  `monkeypatch`/`responses`) повертає правильний cadnum для будинку `19`.
- `_search_kadastrova_karta`: на зафіксованому HTML-фікстурі, як і раніше.
- Опціональний live-smoke тест, позначений `@pytest.mark.live`, не в CI.

### Дашборд

[dashboard.py](../../../dashboard.py) рядок ~352: підпис
`**Кадастрові номери** (БД → kadastr.live)` → `(БД → zem.center)`.

## Завдання 2 — Очистка «Закритої бази» на rieltor.ua

### Мета

Новий CLI-скрипт + кнопка в дашборді, які логіняться в my.rieltor.ua,
відкривають «Закриту базу» й **видаляють усі об'єкти поспіль** у циклі
(на сайті видалення лише поодинці). Перенесення в «Закриту базу» вже зроблено
вручну/сайтом — скрипт лише масово видаляє.

### URL розділу

«Закрита база» = `https://my.rieltor.ua/offers/management?page=1&limit=25&mode=-30`
(`mode=-30` — фільтр закритої бази). Надано власником.

### Архітектура

Новий модуль `rieltor_handler/closed_base_cleaner.py`:

```python
class ClosedBaseCleaner:
    CLOSED_BASE_URL = "https://my.rieltor.ua/offers/management?page=1&limit=25&mode=-30"

    def __init__(self, session: RieltorSession): ...

    def count(self) -> int: ...
        # навігація на URL, підрахунок об'єктів у списку

    def clean(self, max_count: int | None = None, dry_run: bool = False) -> int:
        # 1. навігація на CLOSED_BASE_URL
        # 2. цикл: знайти перший об'єкт → delete → підтвердити діалог →
        #    дочекатися зникнення/перезавантаження списку → повтор
        # 3. зупинка коли список порожній / досягнуто max_count
        # 4. dry_run=True → лише порахувати, нічого не видаляти
        # повертає кількість видалених
```

Логін — через наявний `RieltorSession` (той самий, що в `RieltorOfferPoster`).
Обробка помилок: `RieltorErrorPageException`, перевірка сесії, лог кожного
видалення в `logger.info`.

### Discovery-крок (перший крок реалізації)

URL відомий, але DOM кнопки видалення й діалогу підтвердження — ні. Реалізація
починається з короткої сесії у **видимому** браузері (creds з `.env`):
зайти на `CLOSED_BASE_URL`, зробити `page.content()` / snapshot, знайти:
- селектор контейнера одного об'єкта в списку;
- селектор кнопки/іконки видалення в межах об'єкта;
- селектор діалогу підтвердження та кнопки «Видалити/Так».
Зашити знайдені селектори в `ClosedBaseCleaner`.

### CLI

[main.py](../../../main.py): нова підкоманда

```bash
uv run python main.py clean-trash [--max-count N] [--dry-run] [--no-headless]
```

Створює `RieltorSession` → `login()` → `ClosedBaseCleaner.clean(...)` →
лог підсумку. Слідує наявному патерну інших підкоманд (`post`, `cadastral`).

### Дашборд

Нова bordered-секція в правій колонці, після «Кадастрові номери»:

```
🗑 Очистити сміття на rieltor.ua
caption: Видаляє всі об'єкти із «Закритої бази» (неудачні/неправильні чернетки).
[ ] Я підтверджую видалення        [Макс. ___]
[🗑 Очистити «Закриту базу»]   (disabled, доки чекбокс не стоїть)
```

- Кнопка `disabled=not confirm_checkbox or proc_is_running(...)`.
- Запуск через наявний `launch(cmd, log_level)` із
  `["uv","run","python","main.py","clean-trash", ...]`, headless з глобального
  тумблера.
- Стан процесу/зупинка/результат — той самий патерн, що `cadastral_proc`
  (`st.session_state.cleanup_proc`).

### Безпека

- Обов'язковий чекбокс підтвердження в дашборді (рішення власника).
- `--dry-run` у CLI (показує кількість, нічого не видаляє) — для тестування.
- `--max-count` — обмеження за один прогін.

### Тести

UI-автоматизація залежить від живого DOM, тому юніт-тести обмежені:
- `ClosedBaseCleaner` парсинг кількості об'єктів на зафіксованому HTML-фікстурі
  (після discovery).
- Логіка циклу (зупинка по `max_count`, по порожньому списку) — на мокнутій
  page-обгортці.
- Ручна перевірка повного циклу на реальному акаунті в headed-режимі.
